## 一、删除笔记(全新能力,建议在 guide 手册加一节)

按**标题**删除已发布笔记(创作中心导出无 note_id,业务主键就是标题)。异步契约,和发布/导出同款:

```bash
# 触发(不可逆!先跟运营确认标题无误)
curl -X POST -H "X-API-Key: $KEY" -H "Content-Type: application/json" \
  -d '{"title": "笔记标题", "count": 1}' \
  https://mcp.nbdpsy.com/api/accounts/<账号id>/note-deletions
# → 202 {"deletion_id": "...", "status": "running"}

# 轮询(每 3-5s,约 1-2 分钟到终态)
curl -H "X-API-Key: $KEY" https://mcp.nbdpsy.com/api/note-deletions/<deletion_id>
# → {"status":"done","deleted":1,"remaining":0}
#   或 {"status":"error","reason":"note_not_found: ..."}
```

要点(写进 guide 的话术照这个说):

- **不可逆**。触发前必须和运营确认账号 + 标题;标题精确匹配(容忍卡片截断省略号)。
- `count` 参数:同标题发重复了(同题多篇)时,一次会话最多删 `count` 篇(≤10),
  `remaining` 回剩余同题篇数。典型用法:重复发布清理,`count` = 多余篇数,留 1 篇。
- 服务端有防误点闸:确认弹窗文案必须含「删除」才会点确认,点错图标会自动 Escape 报错,
  不会误删/误操作。
- `status` 三态:`running` / `done`(带 `deleted` + `remaining`)/ `error`(带 `reason`:
  `note_not_found`=该号没有这标题的笔记;`need_manual_login`=creator 登录态失效需重扫码)。
- deletion_id 是进程内存台账,server 重启即失效(404),重新发起即可。
- 同号的发布/验活/导出/删除共享一把账号锁自动串行——**别对同一账号并发发起**,
  会排队不会坏,但轮询会显得久。

## 二、笔记数据导出(guide ⑥ 已有,补一条新语义)

功能不变(`POST /api/accounts/{id}/note-exports` → 轮询 → `GET /api/accounts/{id}/notes`),
新增一个明确错误语义,guide ⑥ 的说明区建议补一句:

- **`reason: "no_data: 数据看板暂无笔记数据(新发笔记通常次日才入看板),无文件可导出"`**
  —— 账号笔记都是**当天刚发**的时,创作中心数据看板还是空表,没有文件可导。
  这不是故障,等次日看板出数据再拉即可。(旧版此场景误报"download 超时",已修。)
- 运营话术:「今天刚发的笔记,数据明天才能拉到,不用管它。」

## 三、发布链路改进(对 skill 透明,FAQ 可能用到)

运营无需改任何操作,但这几条能解释老问题:

1. **重复发布已根治**:旧版偶发一次发布出 2-3 篇同文(点发布后页面响应慢被误判失败而补点)。
   现在点击一旦被接收绝不补点。若运营报"发重了",先确认是不是 2026-07-23 前的旧事;
   新产生的重复可直接用上面的删除接口清理(`count` = 多余篇数)。
2. **话题是真话题实体了**:正文里的 `#话题` 现在走创作中心下拉精选,发出来是蓝色可点击的
   话题 chip(参与话题流量分发),不再是纯文本。发布时每个话题多花 5-7 秒属正常。
3. 发布/导出/删除全程拟人化操作(反"AI 托管"检测),速度已调优(单次点击 ≤2s)。

## 反馈区(nbdpsy-skills 侧追加)

**2026-07-23 · nbdpsy-skills 侧已全量消费（v1.24.0，commit 见本仓库）**：

1. **已落地**：`publish_note.py` 新增 `--delete-note`（含 count 1–10 客户端预检，与服务端 Field(ge=1,le=10) 对齐）、`--delete-status <deletion_id>`（终态权威复查通道）、`--notes --refresh`（触发导出→轮询→读数完整链路，no_data 按非故障处理）；guide 手册新增「⑧ 删除已发布的笔记」（铁律：触发前复述账号+完整标题+删几篇经运营确认）与 ⑥ 的 no_data 话术；FAQ 补「删错了能恢复吗→不能」「发重了→7-23 已根治，旧的用删除清理」。
2. **反馈①（建议排期）：deletion_id 台账建议落库**。对不可逆操作，客户端有三个只能进 unknown 的场景：轮询期持续网络故障、轮询超时、台账 404。前两者我们做了 `--delete-status` 权威复查兜底；但台账 404（server 重启）后「删了没删」**永久不可查**，只能退到数据看板核对——而删重复恰恰高发于当天（看板 no_data 盲区），此时只剩人工去创作中心确认。台账落库（哪怕只存最近 N 条终态）能把这个盲区闭掉。skill 侧现按最安全语义处置：三场景一律 unknown + 绝不盲目重发。
3. **反馈②（同类小项）**：note-exports 的 export_id 若同为内存台账，同一建议（导出可重跑无害，优先级低于删除）。
4. per-account 锁「别并发」、发重根治时间线（7-23 前旧事）均已写入 guide 话术。
